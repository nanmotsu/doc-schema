## ファイル構成について


リンク先があるものは、ファイル名指定＆テンプレートあり。
お互いにトレーサビリティをとっているため、テンプレートの形を守る。

リンク先のないものは、独立しているもの。
### internal
以下を格納。社内共有用に用いる。

| 種類            | schemaファイル名     | 説明                                           |
| --------------- | -------------------- | ---------------------------------------------- |
| source          | source.json          | 議事録から思い付きまで、改修の起点になる部分   |
| decision        | decision.json        | 決定事項。理由を明記する点で要件定義とは異なる |
| task            | task_impl.json       | 実際の実装作業                                 |
| test            | task_test.json       | テスト作業                                     |
| issue           | issue.json           | 調査が必要な問題点                             |
| spec_internal   | spec_internal.json   | 内部仕様（開発者寄りのロジック・機能仕様）     |
| design_internal | design_internal.json | 開発者向け構成図（内部仕様書で用いる）         |

Sourceの種別は以下

| 種別名 | 内容               |
| ------ | ------------------ |
| MTG    | 会議               |
| DEV    | 実装中に発生       |
| IDEA   | 思いつき           |
| BUG    | 不具合起点         |
| REQ    | 要望               |
| DOC    | 既存仕様からの変更 |


### external
以下を格納。客先共有用に用いる。
変更履歴などは、各ファイルの先頭に記載

| 種類            | schemaファイル名     | 説明                               |
| --------------- | -------------------- | ---------------------------------- |
| requirement     | requirement.json     | 要件定義書。客先との合意で用いる   |
| spec_external   | spec_external.json   | 外部仕様書（客先寄り）             |
| design_external | design_external.json | 客先用構成図（外部仕様書で用いる） |
| glossary        | -                    | 用語定義                           |

### docs

| 種類             | schemaファイル名      | 説明                   |
| ---------------- | --------------------- | ---------------------- |
| setup            | -                     | 環境構築               |
| deploy           | -                     | デプロイ手順           |
| runbook          | -                     | 運用関連               |
| trouble_shooting | trouble_shooting.json | トラブルシューティング |
| manuals          | -                     | マニュアル・規程類     |

assets（生データ、画像、excelやsqlなど）
md側（TASKやDecisionなど）でlinkedで「doc/assets/client/?????????.png」などと書いておく。

| サブフォルダ | 内容                                                   |
| ------------ | ------------------------------------------------------ |
| client       | 客先提供データ                                         |
| test         | テスト用データ                                         |
| analysis     | 分析用データ                                           |
| evidence     | 証拠・根拠データ                                       |
| figures      | 文書に埋め込む図版・画像（`:::figure` で参照するもの） |


マークダウンで書く前提であるが 、内部に配置する画像については、
internal/external/docs内のサブフォルダ内にsrcディレクトリを作成し、
そこに 「各ファイル名_asset_01.png」のように配置していく。

複数ドキュメントで共有する図版・ロゴ・フロー図など、プロジェクト横断で再利用する画像は
`docs/6_assets/figures/` に配置し、`:::figure` ディレクティブの `src=` で参照する。

## 補足

### ※InternalとExternalの違い

| 観点       | internal   | external       |
| ---------- | ---------- | -------------- |
| 目的       | 思考・検討 | 合意           |
| 内容       | 未確定あり | 確定のみ       |
| 見せる相手 | 開発者     | 客先・非技術者 |
| 粒度       | 詳細       | 抽象           |
| 状態       | 変化中     | 安定           |

### ※ ウォーターフォールとの関係性
| 手順     | 対応                    |
| -------- | ----------------------- |
| 要件定義 | external/requirements   |
| 外部設計 | external/specifications |
| 内部設計 | internal/spec           |
| 実装     | internal/tasks          |
| テスト   | internal/tasks / issues |
| 運用     | docs/runbook            |


### ※ ウォーターフォールとアジャイルの違い
| 観点     | ウォーターフォール | アジャイル             |
| -------- | ------------------ | ---------------------- |
| スタート | 要件（external）   | 実装・検証（internal） |
| 進み方   | 一方向             | 反復                   |
| 要件     | 最初に固める       | 後から変わる           |
| リスク   | 変更に弱い         | ブレやすい             |
| 強み     | 合意が強い         | 柔軟                   |
