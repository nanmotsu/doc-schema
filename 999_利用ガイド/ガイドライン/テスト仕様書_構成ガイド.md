# テスト仕様書 構成ガイド

## 概要

テスト仕様書（YAML）の原本はソースコード側リポジトリで管理する。  
このシステム側でテストを発火し、実行後の結果（結果追記済みYAML・エビデンス）をこのシステムに保存する。  
Excel変換・スキーマ検証もこのシステムで実施する。

YAMLスキーマとエビデンスのフォルダ構成は**全プロジェクト共通固定**とし、`000_schema/test/` でスキーマを定義する。

---

## 役割分担

| 場所                           | 内容                                                  |
| ------------------------------ | ----------------------------------------------------- |
| ソースコード側リポジトリ       | テスト仕様書YAML（原本）・テストコード                |
| このシステム（各プロジェクト） | 結果追記済みYAML・エビデンス（スクショ等）・Excel出力 |
| `000_schema/test/`             | YAMLスキーマ定義（全プロジェクト共通）                |
| `.tools/scripts/test/`         | 検証・変換ツール（全プロジェクト共通）                |

---

## テストの流れ

1. ソースコード側リポジトリのYAML（原本）を確認する
2. このシステムからテストを発火する
3. テスト実行後、結果（`pass` / `fail` / `pending`）をYAMLに追記する
4. エビデンス（スクショ等）をこのシステムの `evidence/` に保存する
5. 結果追記済みYAMLをこのシステムの `external/5_test_specs/` に保存する
6. `validate.mjs` でスキーマ検証し、`gen_excel.mjs` でExcel変換する

---

## フォルダ構成

```
{プロジェクト}/
└── external/
    └── 5_test_specs/
        ├── preconditions.yaml              ← 共通前提条件（このシステム側で定義）
        ├── TEST-001_ログイン_正常系.yaml    ← 結果追記済みYAML
        ├── TEST-002_ログイン_異常系.yaml
        ├── TEST-003_患者登録.yaml
        ├── evidence/                       ← エビデンス（スクショ等）
        │   ├── TEST-001-TC001-S01.png      ← ファイル名 = ステップID
        │   ├── TEST-001-TC001-S02.png
        │   └── ...
        └── output/                         ← 生成物（Excel）
            ├── TEST-001_ログイン_正常系.xlsx
            └── ...
```

## YAML スキーマ設計

### 共通前提条件ファイル（preconditions.yaml）

```yaml
# external/5_test_specs/preconditions.yaml

preconditions:
  - テストユーザー（test@example.com）が登録済みであること
  - サーバーが起動していること
  - ステージング環境へのアクセス権があること

environment:                    # 実行環境。省略可
  os: Windows 11
  browser: Chrome 124
  server: https://staging.example.com
  db: PostgreSQL 15

test_data:                      # 省略可。全テスト共通の事前データを書く場合に使う
  - label: 共通施設マスタ
    description: 全テストで前提とする施設データ
    columns: [施設ID, 施設名]
    rows:
      - [1, "テスト施設A"]
      - [2, "テスト施設B"]
```

### テスト仕様書本体（TEST-XXX_*.yaml）

```yaml
# external/5_test_specs/TEST-001_ログイン_正常系.yaml

id: TEST-001
title: ログイン機能テスト仕様書（正常系）
version: "1.0"
date: "2026-05-19"
author: 担当者名

preconditions:                  # このテスト仕様書固有の前提条件（省略可）
  - ログイン画面が表示されていること
  - テストユーザーが登録済みであること

test_data:                      # 事前に用意しておくデータ（Excel に表として出力される）
  - label: テストユーザー
    description: テストで使用するアカウント（事前にDBへ登録しておく）
    columns: [メール, パスワード, ロール]
    rows:
      - ["test@example.com", "Password123!", "管理者"]
      - ["user@example.com", "Password123!", "一般ユーザー"]
  - label: 施設マスタ
    columns: [施設ID, 施設名]
    rows:
      - [1, "テスト施設A"]

cases:
  - id: TC-001
    title: 正常ログイン
    steps:
      - id: TEST-001-TC001-S01
        subtitle: メールアドレス入力                                            # 試験項目・小見出し（省略可）
        precondition: ログイン画面が表示されていること
        action: メールアドレスを入力する
        expected: 入力フィールドに値が反映される
        evidence:
          type: automated
          testFile: C:/repos/blueberry-app/tests/Feature/Auth/LoginTest.php  # ソースコード側の絶対パス
          screenshot: TEST-001-TC001-S01.png                                  # evidence/ フォルダ内のファイル名

      - id: TEST-001-TC001-S02
        subtitle: パスワード入力
        action: パスワードを入力する
        expected: マスク表示される
        evidence:
          type: automated
          screenshot: TEST-001-TC001-S02.png

      - id: TEST-001-TC001-S03
        subtitle: ログイン実行
        action: ログインボタンをクリックする
        expected: ダッシュボードへ遷移する
        evidence:
          type: manual
          screenshot: TEST-001-TC001-S03.png

    result: pass   # pass / fail / pending

  - id: TC-002
    title: パスワード誤りでログイン失敗
    steps:
      - id: TEST-001-TC002-S01
        subtitle: パスワード誤りでのログイン試行
        precondition: ログイン画面が表示されていること
        action: 誤ったパスワードを入力してログインする
        expected: エラーメッセージが表示される
        evidence:
          type: manual
          screenshot: TEST-001-TC002-S01.png

    result: pass

  - id: TC-003
    title: 未登録メールでログイン
    steps:
      - id: TEST-001-TC003-S01
        subtitle: 未登録メールでのログイン試行
        action: 未登録のメールアドレスでログインする
        expected: 「メールアドレスが見つかりません」と表示される
        evidence:
          type: none

    result: pending
```

**フィールド定義（トップレベル）:**

| フィールド      | 必須 | 説明                                                   |
| --------------- | ---- | ------------------------------------------------------ |
| `id`            | ✓    | `TEST-{連番}` 形式。全体で単一連番                     |
| `title`         | ✓    | テスト仕様書のタイトル                                 |
| `version`       | ✓    | バージョン（文字列）                                   |
| `date`          | ✓    | 作成日                                                 |
| `author`        | ✓    | 担当者名                                               |
| `preconditions` | —    | このテスト固有の前提条件リスト（文字列の配列）。省略可 |
| `test_data`     | —    | 事前データ定義の配列（後述）。省略可                   |
| `cases`         | ✓    | テストケースの配列                                     |

**test_data フィールド定義:**

| フィールド    | 必須 | 説明                                    |
| ------------- | ---- | --------------------------------------- |
| `label`       | ✓    | データセットの名称（Excelの表タイトル） |
| `description` | —    | 補足説明。省略可                        |
| `columns`     | ✓    | 列名の配列                              |
| `rows`        | ✓    | データ行の配列（各行も配列）            |

**ステップ フィールド定義:**

| フィールド            | 必須 | 説明                                                                                |
| --------------------- | ---- | ----------------------------------------------------------------------------------- |
| `id`                  | ✓    | `{TEST番号}-TC{連番}-S{連番}` 形式。全体で一意                                      |
| `subtitle`            | —    | 試験項目・小見出し。`cases[].title` の下位分類。省略可                              |
| `precondition`        | —    | このステップ固有の前提条件。省略可                                                  |
| `action`              | ✓    | 操作内容                                                                            |
| `expected`            | ✓    | 期待結果                                                                            |
| `evidence.type`       | ✓    | `automated` / `manual` / `none`                                                     |
| `evidence.testFile`   | —    | `automated` 時：ソースコード側テストコードの絶対パス                                |
| `evidence.screenshot` | —    | `automated` / `manual` 時：`evidence/` フォルダ内のファイル名（`{ステップID}.png`） |
| `result`（ケース）    | ✓    | `pass` / `fail` / `pending`                                                         |

## スキーマ定義

YAMLスキーマは全プロジェクト共通で `000_schema/test/` に JSON Schema 形式で定義する。  
`validate.mjs` はこのスキーマを参照して検証する。

```
000_schema/
└── test/
    ├── test_spec.json       ← テスト仕様書YAMLのスキーマ
    └── preconditions.json   ← 共通前提条件YAMLのスキーマ
```

## ツール構成（将来実装）

```
.tools/scripts/
└── test/
    ├── validate.mjs      # YAMLスキーマ検証（000_schema/test/ を参照）
    └── gen_excel.mjs     # YAML → Excel 変換（客先提出用）
```

### validate.mjs

- 対象: `external/5_test_specs/*.yaml`（`preconditions.yaml` は別スキーマで検証）
- スキーマ: `000_schema/test/test_spec.json`
- チェック内容: 必須フィールドの存在、IDフォーマット（`TEST-\d{3}-TC\d{3}-S\d{2}`）、`evidence.type` の値

### gen_excel.mjs

- 対象: `external/5_test_specs/*.yaml`（`preconditions.yaml` を除く）
- 出力: `external/5_test_specs/output/{ファイル名}.xlsx`
- Excel の各ファイルは以下の **3シート構成**:
  1. **共通前提条件** — `preconditions.yaml` の内容（`preconditions` リスト + `environment` 表 + `test_data` 表）
  2. **事前データ** — `preconditions` リスト + `test_data` の各テーブル
  3. **テストケース** — ステップ一覧

## 役割まとめ

```
ソースコード側リポジトリ
  テスト仕様書YAML（原本）  ← 仕様定義の場所
  テストコード              ← 自動テストの実装

このシステム（各プロジェクト）
  external/5_test_specs/
    *.yaml                  ← 結果追記済みYAML（ソースコード側から持ってきて結果を追記）
    evidence/               ← エビデンス（スクショ等）
    output/                 ← Excel変換結果

000_schema/test/            ← YAMLスキーマ定義（全プロジェクト共通）
.tools/scripts/test/        ← validate.mjs・gen_excel.mjs（全プロジェクト共通）
```

- `evidence.screenshot` は `evidence/` フォルダ内のファイル名のみ記載。絶対パス不要。
- `evidence.testFile` はソースコード側の絶対パスで参照（読み取り専用・変更なし）。
- TEST番号は全体で単一連番のため、どのYAMLからでもステップIDが一意に特定できる。
- `evidence.type: none` を明示することで「エビデンスなし」も記録として残せる。
